# PJ-AG4 项目计划

## 一、当前判断

项目的仿真主链路已经能跑，基础能力不缺，短板主要在三块：

- 结果产出偏弱。现在主要是 `simulation_results.csv` 加一张汇总图，适合调试，不够适合直接交作业、写报告或做展示。
- 配置入口偏硬编码。市场参数和 Agent 参数主要还放在默认配置里，做场景切换和对照实验不够顺手。
- 实验和 LLM 能力已经有雏形，但还不够稳，也不够好用。尤其是长轮次 LLM 运行、批量实验入口、决策解释这几块，离“可交付”还差一层。

接下来不建议继续堆底层机制，先把输出、配置、实验入口和稳定性补齐，把项目从“能跑”拉到“能交、能比、能讲清楚”。

## 二、优先级排序

### P0：实验报告生成器

目标：把现在的 CSV 和单张 PDF 图升级成完整报告产物，至少支持 Markdown，后续可扩展到 HTML。

报告内容至少包含：

- 运行参数与场景说明
- 每轮价格、库存、声誉、利润变化
- 关键事件摘要
- 最终赢家与原因分析
- 各 Agent 策略表现对比
- 可直接引用到课程报告里的结论段落

这项优先级最高。原因很简单，仿真引擎本身已经有了，最缺的是把结果整理成能直接交出去的材料。

### P1：轻量看盘 Dashboard

目标：提供一个本地可运行的分析面板，把单次运行和多次实验结果直接可视化，方便演示、复盘和对比。

这个 Dashboard 不走重前端路线，先做轻量版本，重点是分析效率，不是页面包装。更合适的形态是本地分析面板，而不是完整 Web 产品。

第一版建议支持：

- 每轮价格、库存、声誉、利润曲线
- 三个 Agent 的动作对比
- 关键事件时间线
- 单次运行详情查看
- 多 seed、多策略结果切换
- heuristic 和 LLM 模式结果对照

这项适合放在报告生成器之后做。原因是报告生成器会先把摘要逻辑、关键事件抽取、赢家原因分析这些底层能力定下来，Dashboard 只需要复用这些结果，不必重新发明一套展示数据。

### P1：配置化场景系统

目标：把 market 和 agent 相关参数从代码默认值中抽出来，支持通过配置文件切换场景。

第一批场景建议先做：

- `baseline`
- `price-war`
- `supply-shock`
- `high-volatility`

格式优先考虑 `TOML`，和当前 Python 项目更贴近，也方便手写和版本管理。后面如果确实有必要，再补 `YAML`。

这项完成后，后续做论文、课堂展示、参数对比会轻很多，也能让 README 和 CLI 更清楚。

### P1：多次运行对比 CLI

目标：给 `quant/` 外面包一层正式入口，做成一键跑 benchmark 的命令。

建议新增：

- `pj-ag4-benchmark`

基础能力包括：

- 多 seed 批量运行
- 多策略对比
- 自动输出排行榜
- 自动输出汇总 CSV 和 Markdown 报告

现在 `quant/` 已经有底子，但使用方式还是偏开发者脚本。这个入口补上后，项目整体完成度会明显提高。

### P1：可解释决策日志

目标：每个 Agent 每轮除了给出动作，还要留下结构化决策说明。

最小可用版本建议包括：

- forecast 为什么这么估
- price 为什么这样定
- quantity 为什么这样补
- risk gate 为什么放行或收紧

这部分即使在 heuristic 模式下也应该有，不能只给 LLM 模式做。因为展示价值和分析价值都很高，而且有助于定位策略异常。

### P2：LLM 长轮次稳定性增强

目标：让 LLM 模式不只是能跑通 smoke test，而是真能支撑多轮实验。

优先补这些：

- 超时重试
- 局部缓存
- 断点续跑
- fallback 轮次显式标记
- 失败日志更清楚

这部分偏工程性，不一定最显眼，但很关键。后面要做长轮次、对照实验或者报告复现，这块迟早得补。

### P2：更丰富的市场事件系统

目标：让市场波动不只来自时间序列本身，而是能出现更像真实市场的外部事件。

第一批事件建议：

- 政策冲击
- 竞品故障
- 供应链断裂
- 大客户集中下单
- 品牌危机

这项能显著提高场景表现力，但不建议现在先做太深。最好等配置化场景和报告系统落地后，再把事件系统接进去，否则容易把复杂度拉高、产出却还跟不上。

## 三、建议推进顺序

### 第一阶段：先把结果做成能交付的东西

先完成：

1. 实验报告生成器
2. 可解释决策日志

这两项一旦落地，项目就不再只是“能跑出一份 CSV”，而是能讲清楚谁赢了、为什么赢、每轮发生了什么。

### 第二阶段：把展示层和实验入口补齐

然后完成：

1. 轻量看盘 Dashboard
2. 配置化场景系统
3. 多次运行对比 CLI

这一阶段完成后，项目会从“手动跑实验”变成“可以直接展示、方便切换场景、方便做横向比较”。

### 第三阶段：补稳定性和复杂场景

最后完成：

1. LLM 长轮次稳定性增强
2. 更丰富的市场事件系统

原因是这两项都很有价值，但都建立在前两阶段已经把入口、配置、结果产物整理好之后。否则就会出现底层越来越复杂，但输出还是不好用的问题。

## 四、近期执行清单

接下来建议按下面顺序推进：

1. 先把单次仿真的 Markdown 报告生成功能做出来，报告里带表格、关键事件和结论摘要。
2. 在 Agent 决策链里补结构化解释字段，并把解释一起写进输出产物。
3. 做一个轻量本地 Dashboard，把曲线、关键事件、结果对比先展示起来。
4. 增加场景配置文件加载能力，先支持 `baseline`、`price-war`、`supply-shock`、`high-volatility`。
5. 增加 `pj-ag4-benchmark` 命令，直接消费 `quant/` 现有能力。
6. 给 LLM 模式补缓存、重试、续跑和失败标记。
7. 最后再把事件系统接入场景配置、报告系统和 Dashboard。

## 五、阶段完成标准

### 第一阶段完成标准

- 单次运行后，除了 CSV 和图，还能生成一份可直接阅读的 Markdown 报告。
- 报告里能看出每轮变化、关键事件、赢家原因和策略差异。
- Agent 的决策解释能够落盘，而不是只能看最终动作。

### 第二阶段完成标准

- 有一个本地可运行的 Dashboard，可直接查看单次运行和对比结果。
- 场景可以从配置文件切换，不需要改 Python 默认值。
- Benchmark 可以通过正式 CLI 跑起来，不需要手写 Python 片段。
- 多策略、多 seed 对比结果有统一输出目录和统一汇总格式。

### 第三阶段完成标准

- LLM 模式跑长轮次时，失败可恢复、回退可见、问题可定位。
- 市场事件可以通过场景配置启用，并进入报告摘要和 Dashboard 时间线。
- 整个项目能支撑一组完整的对照实验和结果展示。
